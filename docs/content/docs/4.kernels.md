### What Is a Kernel?

A `kernel` is a patch that runs one sample at a time instead of one block at a time. It shares the `patch` body grammar — virtual ports, scopes, connections, a sink — but the whole body is compiled into a single per-sample subgraph and exposed to the rest of your graph as one ordinary node.

The reason to reach for one: **feedback**. The block-rate graph is topologically sorted, so a cycle is either illegal or costs a full block of latency through a shared delay line. Inside a kernel, cycles are legal and cost exactly one sample.

```rust
// y[n] = x[n] + fb * y[n - delay]: a feedback comb, impossible at block rate
kernel comb(fb = 0.6) {
    in audio_in

    audio {
        add { val: 0.0 },
        tap { delay_length: 5, chans: 1 },
        mult { val: $fb }
    }

    audio_in >> add[0]
    add >> tap
    tap >> mult[0]
    mult >> add[1]     // closes the cycle

    { add }
}

patches {
    comb: c {}
}

audio {
    sine { freq: 330.0 }
}

sine >> c.audio_in

{ c }
```

From the outside, `c` is just a node: port selection, `* N` spawning, selectors, and mixing with block-rate nodes all work unchanged. The block adaptation happens automatically (the same `PerSample` adapter you would use for a custom per-sample node).

### How Feedback Works

There is no explicit delay element. Each interior node writes its output into a persistent value table, and nodes execute in a topological order computed by a depth-first search that skips cycle-closing (back) edges:

- A **forward** edge's source has already run this sample, so the reader sees the *current* sample.
- A **back** edge's source runs later, so the reader sees the *previous* sample — an implicit z⁻¹.

Every cycle gets exactly one implicit z⁻¹. Which edge of a cycle carries it follows declaration order (the search starts from nodes in the order they appear in the kernel body), so if the placement matters to your design, order your declarations so the loop "closes" where you want the delay.

### Kernel-Capable Nodes

Kernels dispatch through a closed enum rather than dynamic dispatch, so only nodes with a per-sample implementation are allowed inside a body: `sine`, `saw`, `svf`, `onepole`, `allpass`, `tap`, `map`, and the ops (`mult`, `add`, `sub`, `div`, `gain`). Anything else fails at build time with a clear error.

Every kernel-capable node is verified by an equivalence suite: driving it sample-by-sample must produce the same output as its block-rate `process` (bit-exact for the scalar nodes, within a small tolerance for the SIMD oscillators).

### Performance: When to Graduate to a Custom Node

Kernels trade speed for usability, deliberately. Each sample, every interior node pays an enum dispatch plus a wiring gather, and nothing can use SIMD across the block. The `Plate reverb` benchmark (`cargo bench --bench nodes -- Plate`) measures the same Lexicon-480L-style plate both ways — handwritten Rust versus the ~66-node reference kernel:

| Implementation | 4096-sample stereo block @ 48kHz | Share of realtime budget |
|---|---|---|
| Custom Rust node (`plate480`) | ~315 µs | ~0.4% |
| Kernel DSL (`PLATE_KERNEL`) | ~2.6 ms | ~3% |

That is roughly an **8× penalty** — or, viewed per node, about 10 ns of overhead per node per sample. Both versions are far inside the realtime budget, and their output envelopes match within a fraction of a percent.

Practical guidance:

- **Prototype in a kernel.** Topology changes are a text edit; you hear the result immediately. A reverb-sized kernel costs a few percent of one core.
- **Stay in a kernel** if the graph fits your budget. A handful of moderate kernels is fine; the overhead only matters when it is multiplied.
- **Graduate to a custom Rust node** (see [Custom Nodes](/docs/custom-nodes)) when the topology has settled and one of these is true: you are spawning many instances (`* 16` voices of a kernel multiplies its cost 16×), the kernel has grown into your realtime budget, or the inner loop would benefit from SIMD/hand fusion. Implement `PerSampleNode` if you still need single-sample feedback (wrap it in `PerSample`), or `Node` directly for block processing.

The intended workflow is exactly what happened with the plate: sketched as a kernel, verified against a reference, then the hot version lives in Rust while the kernel remains the readable specification.

### The Reference Kernel

`legato::kernel::PLATE_KERNEL` is a full plate reverb (figure-eight feedback tank, modulated allpasses, 14-tap output matrix) authored as a kernel. Prepend it to any graph source and instantiate `plate`:

```rust
let graph = format!("{PLATE_KERNEL}\n patches {{ plate: verb {{ decay: 0.6, wet: 0.5, dry: 0.5 }} }} ...");
```

It doubles as the engine's stress test: an RMS-envelope equivalence test pins it against the handwritten `plate480` node within a few percent.

### Current Limitations

- Kernel bodies may only contain kernel-capable leaf nodes — no nested patches or kernels, no `* N` spawning, selectors, pipes, or port slices *inside* the body.
- No parameter arithmetic in the DSL (e.g. you cannot derive `1 - $mix`; expose `wet` and `dry` separately instead).
- Template substitution replaces whole values: `new_range: $range` works (including when `$range` is an array like `[-1.0, 1.0]`), but per-element templates inside an array literal (`[$lo, $hi]`) do not substitute.
- No message routing to interior kernel nodes yet — set kernel behavior through params at instantiation.
